# 解決したい課題

既存のカレンダーは、

- 予定は管理できる
- 実際に何をやったかは残らない

ことが多い。

また、Trelloやタスク管理ツールは、

- タスク管理はできる
- いつやるかが分かりづらい

という課題がある。

本アプリでは、

- 予定
- タスク
- 作業実績
- 振り返り

を統合して管理する。

# 自作カレンダーアプリ 要件定義書

- 作成日: 2026-06-21
- バージョン: v0.1（MVP draft）
- ステータス: ドラフト

---

## 1. 背景・目的

### 1.1 背景
個人の自作サービス群（エコシステム）を拡張する一環として、カレンダーアプリを開発する。
既に「習慣トラッカー」が稼働しており、今後も自作サービスが追加されていく前提に立つ。

### 1.2 目的
本カレンダーアプリは、単体のスケジュール管理ツールとしてではなく、
**エコシステム内の各サービスが生成するイベントを集約表示する「ハブ」**として位置づける。

将来的には、カレンダー上での操作が他サービスへフィードバックされる
**双方向連携**も視野に入れるが、MVPではスコープ外とする。

### 1.3 本ドキュメントの位置づけ
- MVP（個人利用・最速リリース）を最優先のゴールとする
- ただし、将来の拡張（複数サービス連携、双方向連携）を阻害しない設計とする
- 「今やること」と「今はやらないが構造だけ用意すること」を明確に分離する

---

## 2. 全体構成（前提・確定事項）

| 項目 | 内容 |
|---|---|
| フロントエンド | Next.js 16 (App Router) / TypeScript / Zustand / Tailwind CSS / shadcn-style UI |
| ホスティング形態 | 静的エクスポート（`output: 'export'`） |
| 認証 | Amazon Cognito（**既存ユーザープールを共用**、新規構築なし） |
| API | Amazon API Gateway + AWS Lambda |
| データストア | Amazon DynamoDB |
| 既存連携先 | 習慣トラッカー（同一Cognitoユーザープール） |

### 2.1 アーキテクチャ概要図（テキスト）

```
[Browser (Next.js static export)]
        │  Cognito Hosted UI / SDK (Auth)
        ▼
[Amazon Cognito User Pool] ←─ 既存・共用
        │ JWT
        ▼
[API Gateway] ── [Lambda: Calendar API] ── [DynamoDB: CalendarEvents]
        │
        └── [Lambda: Aggregator] ──(PULL)──→ [習慣トラッカー API]
                                  ──(将来PUSH)──→ [DynamoDB: ExternalEvents]
```

## 3. スコープ定義

### 3.1 MVPスコープ（今回作る）
- カレンダー本体のCRUD（月表示・週表示・日表示、イベント作成/編集/削除）
- Cognito認証（既存ユーザープール利用、ログイン/ログアウト）
- 習慣トラッカーのイベントをPULL方式で取得し、カレンダー上に集約表示
- イベントの出典（source）によるフィルター表示（ON/OFF切り替え）
- レスポンシブ対応（PC/モバイル）

### 3.2 将来スコープ（Phase 2以降・今回は実装しないが設計上ブロックしない）
- 外部サービスからのPUSH型イベント登録API（DynamoDB直接書き込み）
- カレンダー側操作の他サービスへの書き戻し（双方向連携）
- 複数の外部サービス（3つ目以降のアプリ）との連携
- 通知機能（リマインダー、Push通知）
- 共有・コラボレーション機能（複数ユーザー間のカレンダー共有）

### 3.3 明確に対象外（今回もPhase 2でも未定）
- 他社カレンダー（Google Calendar等）との連携
- マルチテナント/マルチユーザー向けSaaS化

## 4. 機能要件

### 4.1 認証・認可
| ID | 要件 |
|---|---|
| AUTH-01 | 既存Cognitoユーザープールを用いたログイン/ログアウトができる |
| AUTH-02 | JWT（IDトークン）をAPI Gatewayへの全リクエストに付与し、Lambda側で検証する |
| AUTH-03 | 未ログイン状態でカレンダー画面にアクセスした場合、ログイン画面にリダイレクトする |

### 4.2 カレンダー基本機能
| ID | 要件 |
|---|---|
| CAL-01 | 月表示・週表示・日表示の切り替えができる |
| CAL-02 | イベントの新規作成（タイトル、開始/終了日時、終日フラグ、メモ）ができる |
| CAL-03 | イベントの編集・削除ができる |
| CAL-04 | 繰り返しイベント（毎日/毎週/毎月）を作成できる（MVPでは簡易実装でも可） |
| CAL-05 | 自社（カレンダー自身）のイベントと外部サービス由来のイベントを区別なく同一カレンダー面に表示する |

### 4.3 外部イベント集約（ハブ機能）— 本アプリの独自要素・中核
| ID | 要件 |
|---|---|
| HUB-01 | 習慣トラッカーのイベントAPIをLambda（Aggregator）が定期的/オンデマンドにPULLし、カレンダー表示用に正規化する |
| HUB-02 | 各イベントは `source`（出典識別子。例: `native`, `habit-tracker`）を必ず持つ |
| HUB-03 | UI上でsource別にイベント表示のON/OFFを切り替えられるフィルターを提供する |
| HUB-04 | source別に視覚的な区別（色分け等）を行い、フィルターと連動させる |
| HUB-05 | 外部サービスのAPI障害時にも自社イベント表示は継続できる（外部取得失敗を握り込み、画面全体を落とさない） |

### 4.4 データモデル（将来拡張を見据えた予約）
| ID | 要件 |
|---|---|
| DATA-01 | イベントエンティティに `source`, `sourceEventId`, `externalUrl`（任意）, `syncDirection`（`read-only` / `bidirectional`、将来用）を持たせる |
| DATA-02 | MVPでは `syncDirection` は常に `read-only` 固定とし、将来のPUSH/双方向対応時に値を切り替えられる構造にする |
| DATA-03 | 外部イベントはカレンダー側DynamoDBにキャッシュ保存し、表示の高速化と外部API障害時のフォールバックに用いる |

## 5. 非機能要件

| カテゴリ | 要件 |
|---|---|
| パフォーマンス | 月表示初期ロードを2秒以内（キャッシュ利用時） |
| 可用性 | 外部サービス（習慣トラッカー）障害時も自社カレンダー機能は継続動作 |
| セキュリティ | 全APIエンドポイントはCognito JWT検証必須。CORS設定は自社ドメインのみ許可 |
| 保守性 | 静的エクスポート構成のため、APIキーやシークレットをフロントに含めない |
| 拡張性 | 新規の外部サービスを追加する際、Aggregator Lambdaに「アダプタ」を1つ追加すれば対応できる構造とする |
| コスト | 個人利用規模のため、DynamoDB/Lambdaはオンデマンド課金を基本とする |

### 6.1 テーブル: `CalendarEvents`

| 属性 | 型 | 説明 |
|---|---|---|
| PK | String | `USER#<userId>` |
| SK | String | `EVENT#<eventId>` |

| endAt | String (ISO8601) | 終了日時 |

| createdAt / updatedAt | String (ISO8601) | 管理用 |

## 7. 連携方式の方針（重要・確定事項）

| 観点 | MVP方針 | 将来方針 |
|---|---|---|
| データ取得方向 | カレンダー側がPULL（習慣トラッカーAPIを呼ぶ） | PUSH（習慣トラッカー側がイベント発生時に書き込む）も選択可能に |
| 書き込み方向 | カレンダー→外部への書き戻しは行わない | カレンダー操作を外部サービスAPIに反映（双方向） |
| 理由 | PULLはカレンダー単体の改修で完結し、外部サービスの改修不要。最速MVPに最適 | 双方向化にはイベント設計・競合解決（コンフリクト処理）が必要なため、別途設計フェーズを設ける |

## 10. 用語定義

| 用語 | 定義 |
|---|---|
| ハブ | 複数の自作サービスのイベントを集約表示するカレンダーの役割を指す本ドキュメント内の呼称 |
| source | イベントの出典元サービスを表す識別子 |
| PULL型連携 | カレンダー側が外部サービスのAPIを呼び出してデータを取得する方式 |
| PUSH型連携 | 外部サービス側がイベント発生時にカレンダー側のデータストアへ書き込む方式 |
| 双方向連携 | カレンダー上の操作が外部サービスにも反映される連携方式（将来要件） |
